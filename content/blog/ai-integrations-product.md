## С чего начать внедрение AI в продукт

С AI часто происходит так: на презентации всё выглядит отлично, а в production — странные ответы, жалобы и растущий счёт за API. Это нормально: **LLM — не фича, а подсистема** с данными, политиками и мониторингом.

Не пытайтесь сразу «добавить AI везде». Выберите один сценарий с измеримой экономикой:

| Сценарий | Типичный эффект | Сложность |
|----------|-----------------|-----------|
| L1-поддержка по FAQ | −20–40% нагрузки на операторов | Средняя |
| Поиск по базе знаний (RAG) | Быстрее онбординг, меньше эскалаций в HR/IT | Средняя |
| Черновики для менеджеров | Экономия времени на рутинных текстах | Низкая |
| Агент с действиями в CRM | Высокий ROI, высокий риск ошибок | Высокая |

Подробнее про RAG — в статье [RAG для базы знаний](/blog/rag-knowledge-base-guide). Roadmap на квартал — [AI roadmap на 90 дней](/blog/ai-product-roadmap-90-days).

## Архитектура: как устроен production-ready AI-контур

Минимальная схема, которую мы используем в проектах DevSystems:

```
Пользователь → API Gateway → Orchestrator
                ↓
         [Policy / PII filter]
                ↓
         Retrieval (vector + keyword)
                ↓
         LLM (prompt + context)
                ↓
         [Guardrails / citation check]
                ↓
         Ответ или эскалация к оператору
```

**Orchestrator** решает: какой промпт, какая модель, нужен ли retrieval, когда вызывать human handoff.

**Policy layer** — до LLM: маскирование ПДн, блокировка запрещённых тем, rate limits.

**Retrieval** — гибридный поиск (embeddings + BM25) по актуальным чанкам; без этого RAG «галлюцинирует уверенно».

**Guardrails** — после LLM: проверка на токсичность, утечку инструкций, соответствие найденным источникам.

## Почему бывают плохие ответы

| Симптом | Вероятная причина | Что делать |
|---------|-------------------|------------|
| Выдуманные факты | Слабый retrieval или пустой контекст | Ужесточить «отвечай только по контексту», fallback «не знаю» |
| Устаревшие ответы | Старые документы в индексе | Версионирование KB, TTL re-index |
| Дорогой каждый запрос | Длинный контекст, top-k слишком большой | Сжатие чанков, rerank top-5, кэш |
| Жалобы на тон | Общий промпт | Отдельные промпты по каналу (чат/email) |

## Пример конфигурации guardrails (псевдокод)

```typescript
async function answer(userMessage: string, sessionId: string) {
  const policy = await checkPolicy(userMessage);
  if (!policy.allowed) return escalate("policy", sessionId);

  const chunks = await retrieve(userMessage, { topK: 5, minScore: 0.72 });
  if (chunks.length === 0) return fallbackNoContext(sessionId);

  const draft = await llm.complete({ message: userMessage, chunks });
  if (!citationsMatch(draft, chunks)) return escalate("low_confidence", sessionId);

  return { text: draft.text, sources: chunks.map((c) => c.id) };
}
```

## Безопасность и compliance

Для FinTech, health и B2B SaaS фиксируем до разработки:

- классификация данных (что можно в cloud LLM);
- хранение логов диалогов (срок, доступ);
- право на удаление персональных данных из индекса;
- аудит промптов и версий моделей.

В [кейсе AI-ассистента поддержки](/cases/ai-support-assistant) — снижение L1-тикетов при handoff к оператору и eval-метриках.

## Метрики продукта и инженерные SLO

**Продукт:** resolution rate, CSAT, время до первого ответа, % эскалаций.

**Инженерия:** p95 latency, error rate, cost per 1000 requests, hallucination rate на golden set (50–200 эталонных вопросов).

Golden set обновляем каждый спринт — без этого «улучшили промпт» не измерить.

## Чеклист перед production

- [ ] Один use-case и владелец метрик  
- [ ] RAG / промпты версионируются в git  
- [ ] Eval на golden set ≥ целевого порога  
- [ ] Эскалация к человеку < 2 кликов  
- [ ] Дашборд cost + quality  
- [ ] Runbook при деградации модели/API  

## План на 30 дней

1. Discovery: 20 интервью + выгрузка топ-50 тикетов.  
2. Подготовка KB: chunking, дедуп, owner обновлений.  
3. PoC на 5–10% трафика с ручным разбором ошибок ежедневно.  
4. Hardening: guardrails, мониторинг, SLA с поддержкой.  
5. Решение go/no-go по цифрам, не по демо.
