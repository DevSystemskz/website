# Зачем нужен API-шлюз

Когда микросервисов становится больше трёх, появляется проблема: как клиенты общаются с ними? Напрямую — сложно. Нужен шлюз, который управляет этим процессом.

## Что делает API-шлюз

- **Маршрутизация** — направляет запросы нужным сервисам. Клиент не знает, где именно находится функция.
- **Авторизация** — проверяет токены, выдаёт доступ к нужным endpoint-ам.
- **Rate limiting** — ограничивает количество запросов от одного клиента.
- **Трансформация** — меняет формат запросов и ответов, если нужно.
- **Кэширование** — хранит частые ответы, чтобы не нагружать сервисы.
- **Мониторинг** — собирает метрики: кто, сколько, как быстро.

## Когда шлюз не нужен

- Максимум 2-3 сервиса и простая логика.
- Все сервисы одного типа (например, только GraphQL).
- Нет внешних клиентов.

## Популярные решения

- **Kong** — самый популярный, много плагинов. Хорошо для большого масштаба.
- **NGINX** — классический веб-сервер, можно использовать как шлюз. Проще для небольших проектов.
- **AWS API Gateway / Azure API Management** — managed-решения облачных провайдеров.
- **Ambassador** — для Kubernetes, основан на Envoy.

## Как внедрить

1. Начните с простой маршрутизации — только перенаправление запросов.
2. Добавьте авторизацию — проверка токенов на шлюзе.
3. Настройте rate limiting — чтобы не завалили.
4. Добавьте мониторинг — метрики важны.

Главное — начать просто и наращивать функциональность по мере необходимости.
