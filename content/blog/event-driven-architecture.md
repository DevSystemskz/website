# Когда нужна событийная архитектура

Когда пользователь нажал кнопку и система должна сделать много всего: отправить email, обновить статистику, создать запись в логах, уведомить других пользователей. Синхронно — медленно. Асинхронно — быстро.

## Сценарии, где это помогает

- **Массовая обработка** — много пользователей делают много действий одновременно.
- **Интеграция систем** — один процесс должен запустить несколько других систем.
- **Отчётность** — строить отчёты можно фоновой задачей, не блокируя пользователя.
- **Microservices** — сервисы общаются через события, а не напрямую.

## Компоненты

- **Message Broker** — Kafka, RabbitMQ, SQS. Принимает события и передаёт их подписчикам.
- **Producer** — сервис, который создаёт событие.
- **Consumer** — сервис, который обрабатывает событие.
- **Event Schema** — формат события, чтобы все понимали друг друга.

## Типичные проблемы

- **Дублирование событий** — обрабатывайте идемпотентно.
- **Порядок событий** — события могут приходить не по порядку. Продумайте логику.
- **Потеря событий** — настройте retry и мониторинг.
- **Сложность отладки** — сложно понять, что произошло. Хорошее логирование обязательно.

## CQRS и Event Sourcing

- **CQRS** — Command Query Responsibility Segregation. Разделение записи и чтения. Для сложных систем с разными требованиями.
- **Event Sourcing** — хранение всех событий, а не только состояния. Позволяет строить «машину времени» для данных.

Событийная архитектура — это не серебряная пуля. Она добавляет сложности, но даёт масштабируемость и надёжность.
